# six6 拆解整理大纲

## 目标

这份文档不是继续扩展 `six6`，而是为“暂停 six6、本地拆解、逐日整理”服务。

核心目标：

1. 识别 `six6` 里真正有价值的部分。
2. 把概念层和生产层彻底分开。
3. 将原本纠缠在一起的流程，拆成两条互不干扰的正式管线。
4. 在最多 7 天内整理出一套可长期维护的新结构。

---

## 当前判断

### six6 的优点

- 思路好：把认知过程拆成模块。
- 文件协议意识强：强调通过文件通信而不是内存耦合。
- `memory / meditation / daydream` 这些模块有真实价值。
- `pulse` 调度思路清晰，适合做自动化入口。

### six6 的问题

- 文档和实现存在割裂。
- “最终入库文件如何生成”没有形成单一权威入口。
- `approved_decisions` 历史数据结构不统一。
- FalkorDB 已经混入多来源、多时代、多协议的数据。
- “认知模块”与“生产级记忆入库”被混在一个叙事里，边界不清。

### 当前结论

`six6` 适合被拆成能力来源，而不适合作为当前生产记忆系统的总入口。

---

## 拆解后的总结构

建议拆成两个主系统，一个共享底层图库，但主节点类型和协议分开。

### 模块 A：记忆提炼工厂

职责：

- 读取原始记忆素材
- 做结构化蒸馏
- 生成候选
- 生成人工审核报告
- 生成核准文件
- 将核准内容正式写入 FalkorDB

特点：

- 强协议
- 强审计
- 强幂等
- 必须人工审核后才能入库

写入目标：

- `(:Memory)`

### 模块 B：幻想乐园

职责：

- 从历史记忆中发散联想
- 生成创意、假说、灵感种子
- 允许高噪声与实验性内容
- 可入图库，但不进入正式记忆主线

特点：

- 高自由度
- 高创造性
- 可弱审核
- 不影响正式记忆层

写入目标：

- `(:Idea)` 或 `(:DreamSeed)`

### 模块 C：图谱层

职责：

- 作为统一存储层
- 允许不同来源节点共存
- 通过标签、关系、节点类型形成连接

推荐节点类型：

- `(:Memory)` 正式记忆
- `(:Idea)` 幻想产物
- `(:Entity)` 实体
- `(:Tag)` 标签

关键原则：

- 不再让幻想产物直接混入正式 `Memory`
- 不再让上游候选文件直接决定最终数据库结构

## 一周整理计划

建议按 7 天推进，每天只处理一个明确模块。

### Day 1：总盘点与冻结

目标：

- 冻结当前 `six6` 的自动运行角色
- 盘点现有文件、脚本、文档、入库链路
- 明确哪些部分保留，哪些部分暂停

当天产出：

- `six6` 现状总览
- 当前脚本清单
- 当前数据流清单
- 风险点列表

当天要回答的问题：

- 哪些 cron / pulse / 自动入口还在运行？
- 哪些脚本还会写 `six6_lab/`？
- 哪些脚本还会写 `approved_decisions/`？
- 哪些脚本还会写 FalkorDB？

### Day 2：记忆提炼工厂

目标：

- 明确定义正式记忆主链
- 确定从 raw memory 到 approved JSONL 的完整流水线

建议子流程：

1. raw memory
2. distilled memory
3. candidates
4. review report
5. approved decisions
6. ingestion

当天产出：

- 正式记忆主链图
- 每一步输入输出定义
- 每一步责任脚本定义

当天要回答的问题：

- 哪一步是自动的？
- 哪一步必须人工？
- 哪一步产出才允许进数据库？

### Day 3：approved JSONL 协议统一

目标：

- 统一 `approved_decisions` 的结构
- 结束历史多格式并存状态

建议确定的最小字段：

- `candidate_id`
- `topic`
- `content`
- `timestamp`
- `category`
- `source`
- `approved_at`

建议保留字段：

- `maturity`
- `source_file`
- `source_section`
- `schema_version`

当天产出：

- `Approved JSONL V2` 协议草案
- 历史兼容规则
- normalizer 规则

当天要回答的问题：

- `id` 和 `candidate_id` 最终如何分工？
- 缺失字段如何补默认值？
- 哪些旧文件需要兼容读取？

### Day 4：FalkorDB Memory 协议

目标：

- 从数据库视角重定义正式 `Memory` 节点
- 明确幂等更新策略

建议正式字段：

- `id`
- `candidate_id`
- `topic`
- `content`
- `timestamp`
- `category`
- `maturity`
- `source`
- `approved_at`
- `source_file`
- `source_section`
- `schema_version`
- `ingested_at`

当天产出：

- `Memory Node V2` 字段表
- 幂等更新规则
- legacy 节点处理原则

当天要回答的问题：

- 数据库主键是不是永远内部生成？
- 更新依据是不是永远以 `candidate_id` 为准？
- 没有 `candidate_id` 的旧数据怎么对待？

### Day 5：幻想乐园独立化

目标：

- 把 `daydream / topic-lab` 从正式记忆主链拆出去
- 给创意类产物单独定义协议和节点类型

建议字段：

- `id`
- `topic`
- `description`
- `source`
- `created_at`
- `status`
- `maturity`
- `tags`

建议节点类型：

- `(:Idea)`

当天产出：

- 幻想乐园的数据结构
- 与正式记忆层的边界定义
- 与 FalkorDB 的关系模型草案

当天要回答的问题：

- Idea 是否允许自动入图库？
- Idea 是否需要人工审核？
- Idea 与 Memory 如何建立关系？

### Day 6：执行器与目录重组

目标：

- 把脚本层重新分组
- 不再沿用 six6 的叙事式模块命名作为生产主入口

建议新目录方向：

```text
memory-pipeline/
  candidates/
  review/
  approved/
  ingestion/

idea-pipeline/
  generation/
  farming/
  graph-sync/

shared/
  protocols/
  graph/
  utils/
```

当天产出：

- 新目录草案
- 每个脚本的新归属
- 保留脚本 / 废弃脚本清单

当天要回答的问题：

- 哪些脚本要迁移？
- 哪些脚本要重写？
- 哪些脚本只保留参考价值？

### Day 7：总收口与迁移路线

目标：

- 把前 6 天的结论收敛成迁移计划
- 明确“先做什么，后做什么”

建议迁移顺序：

1. 停止 six6 自动入库影响
2. 定义 `Approved JSONL V2`
3. 重写 ingestion executor
4. 统一 FalkorDB Memory V2
5. 拆出 Idea 管线
6. 再决定是否保留 six6 的局部能力

当天产出：

- 最终迁移路线图
- 风险列表
- 待开发清单

## 推荐的最终模块划分

如果最终目标是“拆开 six6，保留价值，形成长期可维护结构”，建议整理成下面 6 个模块。

### 1. 原始记忆采集

负责：

- 收集原始对话
- 收集同步文件
- 统一输入源

### 2. 记忆提炼工厂

负责：

- 提炼 `MEMORY.md`
- 生成候选 JSONL
- 生成人审报告

### 3. 决策分流中心

负责：

- 处理批准 / 废弃
- 维护 `approved_decisions/`
- 维护 `deprecated_decisions/`

### 4. FalkorDB 入库执行器

负责：

- 读取核准文件
- 做格式归一化
- 做幂等入库
- 生成入库战报

### 5. 幻想乐园

负责：

- 发散创意
- 养成种子
- 产生灵感图谱

### 6. 图谱治理层

负责：

- 定义节点协议
- 控制正式记忆与幻想节点边界
- 做图谱清理、审计、迁移

## 当前建议

短期内不要做的事：

- 不要继续扩写 `six6` 叙事文档
- 不要让 `topic-lab-seeds` 再直接影响正式记忆入库
- 不要继续容忍 `approved_decisions` 多结构并存
- 不要把旧图库脏数据当作新协议依据

短期内最值得做的事：

1. 先整理模块边界。
2. 再统一协议。
3. 最后重写执行器。

---

## 一句话路线

不是修补 six6，而是把 six6 拆成：

- 一个严肃的正式记忆工厂
- 一个自由的幻想乐园
- 一个受控的图谱入库层

这样后面每条线都能独立演化，互不污染。
